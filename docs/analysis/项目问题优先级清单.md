# AI-POC 项目问题优先级清单

## 目标

本清单用于对当前 `AI-POC` 项目中已识别的问题进行轻重缓急排序，便于后续按阶段修复。

排序原则：

- `P0`：立即处理。不修会导致核心功能失效，或存在明显安全边界问题。
- `P1`：尽快处理。不会立刻阻断系统，但会明显影响正确性、稳定性或敏感信息安全。
- `P2`：中期处理。主要影响部署适配、协作效率和可维护性。
- `P3`：后续清理。以文档、历史包袱、代码整洁度为主。

---

## P0 立即处理

### 1. Nuclei 执行链断裂

**现象**

- API 层已经切换到 `services/nuclei_service.py`
- 但 POC 库执行逻辑仍然依赖已删除的 `services/nuclei_engine.py`

**影响**

- `nuclei` 类型的 POC 在 POC 库中大概率无法执行
- 当前架构存在明显“半迁移”状态，功能不一致

**涉及文件**

- `services/poc_library_service.py`
- `services/nuclei_service.py`
- `api/routes.py`

**优先级理由**

这是明确的功能性故障，属于核心能力不可用，必须先修。

---

### 2. Nuclei 模板路径缺少安全约束

**现象**

- 外部传入的模板路径、文件夹路径会与 `templates_dir` 直接拼接
- 当前实现缺少严格的目录归属校验

**影响**

- 理论上存在目录逃逸风险
- 会影响模板读取、模板扫描相关接口的安全边界

**涉及文件**

- `services/nuclei_service.py`

**优先级理由**

这是安全边界问题，应在功能修复的同时一起封住。

---

### 3. Python POC 直接动态导入执行

**现象**

- 当前通过动态导入方式执行库中的 Python POC
- 代码运行时没有进程隔离或权限隔离

**影响**

- 只要 POC 内容不可信，就可能直接执行任意代码
- 在多人协作、共享库、外部导入场景下风险很高

**涉及文件**

- `services/poc_library_service.py`

**优先级理由**

如果项目只在单人本地研究环境中使用，短期可容忍；但从系统设计上看，这是高风险点，应尽快限制。

---

## P1 尽快处理

### 4. URL 标准化逻辑会丢失路径和参数

**现象**

- 当前 URL 规范化后只保留 `scheme://host/`
- 原始输入中的路径、查询参数会被丢弃

**影响**

- 很多依赖具体路径、接口、查询参数的漏洞验证会失败
- 影响 POC 执行结果的真实性和准确性

**涉及文件**

- `services/poc_library_service.py`

**优先级理由**

这会直接伤害漏洞验证正确率，属于高价值修复项。

---

### 5. Nuclei 流式扫描超时语义不一致

**现象**

- 前端向后端传入了扫描超时参数
- 但流式扫描内部把 Nuclei 自身超时写死成了固定值

**影响**

- 用户配置与实际行为不一致
- 扫描结果可能提前结束，造成误判

**涉及文件**

- `services/nuclei_service.py`
- `frontend/js/nuclei.js`

**优先级理由**

这是典型的“看起来可配，实际不可配”问题，会伤害用户预期和调试效率。

---

### 6. LLM API Key 明文写入本地配置

**现象**

- API Key 会直接持久化到本地 JSON 文件

**影响**

- 本地泄露风险高
- 提交仓库、共享工作目录或打包分发时容易产生敏感信息暴露

**涉及文件**

- `services/llm_service.py`
- `pocs/llm_config.json`

**优先级理由**

虽然是本地项目常见简化做法，但只要进入多人协作或机器共享场景就不安全，应尽快调整。

---

## P2 中期处理

### 7. 前端硬编码 `127.0.0.1:8000`

**现象**

- 前端多个页面和脚本写死了后端地址和 API 文档地址

**影响**

- 换端口、换主机、反向代理、远程访问时容易失效
- 降低部署灵活性

**涉及文件**

- `frontend/js/app.js`
- `frontend/js/library.js`
- `frontend/index.html`
- `frontend/nuclei.html`

**优先级理由**

不影响单机默认使用，但会阻碍后续部署和环境迁移。

---

### 8. 缺少明确的最小可运行环境约束

**现象**

- 当前环境里没有现成虚拟环境
- 依赖是否已安装、如何启动、如何验证运行状态不够收敛

**影响**

- 新人接手和环境复现成本高
- 问题定位时容易混淆“代码问题”和“环境问题”

**涉及文件**

- `requirements.txt`
- `README.md`

**优先级理由**

这不是核心逻辑故障，但会持续消耗协作成本。

---

## P3 后续清理

### 9. 前端保留了旧版 POC 库逻辑

**现象**

- `app.js` 中仍保留旧版库管理实现
- 新版库逻辑已经拆到 `library.js`

**影响**

- 存在重复逻辑和历史包袱
- 增加维护成本，容易误改

**涉及文件**

- `frontend/js/app.js`
- `frontend/js/library.js`

**优先级理由**

短期不一定出事故，但长期会拖慢维护和重构。

---

### 10. 项目文档与当前实现不一致

**现象**

- README 中仍提到已删除模块、旧端点、旧流程
- 部分存储说明和实际实现不一致

**影响**

- 会误导后续开发
- 增加理解成本和排障成本

**涉及文件**

- `README.md`
- `pocs/README.md`

**优先级理由**

不会立刻导致系统故障，但会持续制造认知偏差。

---

## 推荐执行顺序

建议按以下顺序推进：

1. 修复 `nuclei_engine` 到 `nuclei_service` 的迁移收口
2. 为 Nuclei 模板路径增加目录归属校验
3. 限制 Python POC 执行风险
4. 修复 URL 标准化逻辑，保留路径和参数
5. 统一 Nuclei 扫描超时语义
6. 改造 API Key 存储方式
7. 去掉前端硬编码地址
8. 整理运行环境和启动说明
9. 清理前端旧逻辑
10. 更新 README 和相关说明文档

---

## 简版结论

一句话概括：

当前最该做的不是继续加功能，而是先把 **Nuclei 迁移收口、安全边界、执行链正确性** 三件事处理干净；否则项目表面功能很多，但稳定性和可信度不够。
