# 批量检测任务清单

## 目标范围

本次需求目标：

- 新增独立的批量检测能力，不复用现有单次执行接口作为前端循环调用方案
- 批量检测仅在用户明确选择“批量模式”时启用任务队列
- 支持以下三种批量场景：
  - 单个 URL + 多个 POC
  - 多个 URL + 单个 POC
  - 多个 URL + 多个 POC
- 最小任务单元统一定义为：`url + poc_id`
- Python POC 与 Nuclei 在批量任务层统一调度，但保留各自执行器

## 设计原则

### 1. 单次执行与批量执行分离

- 保留现有单次执行接口：
  - `/api/pocs/{id}/execute`
  - `/api/nuclei/scan`
  - `/api/nuclei/scan/stream`
- 新增独立批量任务接口，不让前端自行循环调用单次接口

### 2. 统一任务编排，不统一底层执行器

- `poc_library_service.execute_poc(...)` 继续负责单条 POC 执行
- `nuclei_service.py` 继续负责 Nuclei 扫描执行
- 新增批量任务服务统一调度它们

### 3. 默认简单，批量增强按需展开

- 默认仍然是单 URL 输入框
- 只有切到批量模式，才出现：
  - URL 文件上传
  - 多行 URL 输入
  - POC 多选
  - 批量任务创建与进度视图

### 4. 第一版必须做任务限流

- 批量扫描是笛卡尔积，必须限制：
  - URL 数量
  - POC 数量
  - 总子任务数量
- 创建任务前必须明确展示预计任务数

## 后端任务清单

### 任务 1：设计批量任务数据表

目标：

- 持久化批量任务及其子任务

建议新增表：

- `batch_tasks`
- `batch_task_items`

建议字段：

- `batch_tasks`
  - `id`
  - `mode`
  - `status`
  - `total_items`
  - `completed_items`
  - `success_items`
  - `failed_items`
  - `vulnerable_items`
  - `config_json`
  - `created_at`
  - `started_at`
  - `finished_at`
- `batch_task_items`
  - `id`
  - `task_id`
  - `poc_id`
  - `target_url`
  - `engine_type`
  - `status`
  - `result_json`
  - `error`
  - `started_at`
  - `finished_at`

验收标准：

- 能保存一个批量任务的总览信息
- 能保存每一个 `url + poc_id` 子任务执行结果

### 任务 2：新增批量任务服务

目标：

- 新增统一的批量任务编排层

建议新增文件：

- `services/batch_task_service.py`

职责：

- 校验输入
- 计算任务规模
- 拆分子任务
- 推进任务状态
- 调用现有执行器
- 汇总结果

验收标准：

- 可通过服务层创建任务、执行任务、查询任务

### 任务 3：新增批量任务 API

目标：

- 提供独立批量任务接口

建议接口：

- `POST /api/batch-tasks`
- `GET /api/batch-tasks`
- `GET /api/batch-tasks/{id}`
- `GET /api/batch-tasks/{id}/items`
- `POST /api/batch-tasks/{id}/cancel`

职责：

- 创建任务
- 查询任务列表
- 查询任务详情
- 查询子任务结果
- 取消任务

验收标准：

- 前端不再循环调用单条执行接口做批量扫描

### 任务 4：实现任务拆分逻辑

目标：

- 把用户选择转换成标准子任务列表

规则：

- 最小单元固定为 `url + poc_id`
- 多 URL × 多 POC 使用笛卡尔积
- `manual` 类型 POC 不允许进入批量执行
- 默认仅允许 `verifiable = true` 的 POC

验收标准：

- 能正确生成预期数量的子任务
- 能过滤不允许进入批量执行的 POC

### 任务 5：实现后台执行模型

目标：

- 任务创建后由后台推进，而不是依赖前端一直持有请求

建议：

- 第一版使用进程内后台 worker
- 限制并发数为 `3-5`

执行器复用：

- Python / POC库：`poc_library_service.execute_poc(...)`
- Nuclei：后续可通过统一任务服务接入 `nuclei_service`

验收标准：

- 创建任务后，即使页面刷新，任务仍可继续推进

### 任务 6：加入批量限制和保护机制

目标：

- 防止任务规模失控

建议限制：

- 最大 URL 数量
- 最大 POC 数量
- 最大子任务数量

建议行为：

- 创建任务前先返回预估任务规模
- 超限直接拒绝

验收标准：

- 用户无法提交超大批量任务拖垮服务

## 前端任务清单

### 任务 7：在 POC 库页增加“批量模式”入口

目标：

- 在现有 POC 库页上扩展批量能力，不单独重做一套页面

建议交互：

- 增加模式切换：
  - 单次检测
  - 批量检测

行为：

- 默认显示单次检测 UI
- 切换到批量检测后显示批量相关控件

验收标准：

- 单次体验不受影响
- 批量功能只在用户主动选择时展开

### 任务 8：在 POC 列表中增加复选能力

目标：

- 支持多个 POC 的筛选和选择

建议交互：

- 可验证 POC 卡片前增加复选框
- 批量操作条显示：
  - 已选数量
  - 全选当前筛选结果
  - 清空选择

说明：

- “全选”第一版只定义为“全选当前筛选/搜索结果”
- 不做全库无限量全选

验收标准：

- 用户可以先筛选，再批量选择一组 POC

### 任务 9：增加批量 URL 输入能力

目标：

- 支持批量目标输入

建议方式：

- `txt` 文件上传
- 多行文本输入框

不建议作为主方案：

- 无限动态加行表单

原因：

- 少量 URL 可用
- 大量 URL 体验差，维护复杂

验收标准：

- 用户可通过上传或粘贴快速导入一批 URL

### 任务 10：增加任务创建确认区

目标：

- 提交前向用户明确展示本次任务规模

建议展示：

- URL 数量
- POC 数量
- 预计子任务总数
- 实际参与批量执行的 POC 数

验收标准：

- 用户在点击“开始批量检测”前能明确知道本次任务规模

### 任务 11：增加任务列表和任务详情视图

目标：

- 展示批量任务进度和结果

建议展示：

- 任务列表：
  - 状态
  - 创建时间
  - 总任务数
  - 已完成数
  - 命中数
- 任务详情：
  - 子任务列表
  - 支持按 URL、POC、状态筛选

验收标准：

- 页面刷新后仍可查看历史任务与当前进度

## Nuclei 接入策略

### 当前判断

- Nuclei 现有批量能力不是框架内建任务队列
- 是当前项目对 Nuclei CLI 的包装调用
- 因此可以纳入统一任务队列体系

### 接入原则

- 不改 Nuclei CLI 本身
- 不强改 `nuclei_service.py` 的底层扫描逻辑
- 统一的是“批量任务编排层”

### 第一版建议

- 先让 POC 库批量任务跑通
- 任务模型中预留 `engine_type`
- 后续再把 Nuclei 独立扫描页接入同一任务体系

## 建议开发顺序

1. 设计数据库表结构
2. 实现 `batch_task_service`
3. 实现批量任务 API
4. 打通后端任务拆分和执行
5. 改 POC 库前端为批量模式
6. 增加 URL 上传/多行输入
7. 增加任务列表和任务详情页
8. 整理 Nuclei 接入点

## 第一版暂不做

- `manual` 类型 POC 批量执行
- 全库无限量全选
- 高级重试策略
- 分布式 worker
- websocket 实时推送
- 批量任务优先级调度
- 导出复杂报表

## 审核重点

你需要重点确认以下几点：

- 是否同意“批量检测必须是独立 API”
- 是否同意“最小任务单元是 `url + poc_id`”
- 是否同意“默认只允许可验证 POC 进入批量任务”
- 是否同意“第一版只做文件上传 + 多行 URL 输入，不做无限动态表单”
- 是否同意“第一版先打通 POC 库批量任务，Nuclei 后接”
