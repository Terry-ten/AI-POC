# 扩展能力任务清单

## 目标原则

- 保持当前核心路径不变：用户默认仍然是“选择 POC -> 输入 URL -> 点击验证”。
- 扩展能力按渐进式接入，不一次性把前端和执行层做复杂。
- 优先建设“平台能力”，再放开 Prompt，让大模型稳定利用这些能力。
- 所有新增能力都要兼容后续三种执行模式：
  - `url_only`
  - `url_with_params`
  - `manual_guide`

---

## 第一阶段：执行模型留好扩展口

### 1. 定义统一执行模式元数据

目标：
- 给每条 POC 增加轻量执行描述，不改变当前默认体验。

建议字段：
- `execution_mode`
  - `url_only`
  - `url_with_params`
  - `manual_guide`
- `verification_method`
  - `direct`
  - `oob`
  - `manual`
- `input_schema`
  - 仅 `url_with_params` 时使用

影响文件：
- `models/schemas.py`
- `services/poc_library_service.py`
- `api/routes.py`
- `pocs/poc_library.db` 迁移逻辑

验收标准：
- 旧记录不报错，默认回落到现有行为。
- 新记录可以带执行模式元数据。
- 当前前端默认仍然只展示 URL 输入框。

优先级：
- P0

---

## 第二阶段：接入外带验证能力

### 2. 新增 OOB 服务层

目标：
- 接入 Interactsh / CEye，支持无回显漏洞验证。

建议实现：
- 新增 `services/oob_service.py`
- 封装：
  - 申请唯一标识
  - 查询轮询结果
  - 统一返回 `命中 / 未命中 / 异常`

参考来源：
- `D:\development\pycharmPRO\pocsuite3\pocsuite3\modules\interactsh\__init__.py`
- `D:\development\pycharmPRO\pocsuite3\pocsuite3\modules\ceye\__init__.py`

影响文件：
- `services/oob_service.py`（新增）
- `services/poc_library_service.py`
- `api/routes.py`
- `frontend/index.html`
- `frontend/js/app.js`

验收标准：
- 平台可统一配置 Interactsh / CEye。
- POC 执行层可调用 OOB 能力完成验证。
- 不需要用户额外输入参数时，仍保持 URL-only 体验。

优先级：
- P0

### 3. 调整 Prompt 以支持 OOB 验证

目标：
- 让模型在适合的无回显漏洞场景中优先生成 OOB 型 POC。

调整方向：
- 对 SSRF / XXE / 无回显 RCE 等场景，允许优先采用外带验证。
- 明确成功标准可以是 OOB 平台回连，而不是页面响应内容。
- 要求优先调用平台封装的 OOB helper，而不是手写外带逻辑。

影响文件：
- `prompts/poc_generation.yaml`
- `services/llm_service.py`

验收标准：
- 模型能在无回显漏洞场景生成 OOB 验证型 POC。
- 不影响当前常规直连型 POC 生成。

优先级：
- P0

## 第三阶段：补结果交付能力

### 4. 增加批量检测报告导出

目标：
- 批量任务完成后，可导出 HTML / TXT / JSON 报告。

参考来源：
- `D:\development\pycharmPRO\pocsuite3\pocsuite3\plugins\html_report.py`
- `D:\development\pycharmPRO\pocsuite3\pocsuite3\plugins\file_record.py`

建议实现：
- 导出任务级报告
- 至少包含：
  - 任务基本信息
  - URL 数量
  - POC 数量
  - 命中 / 未命中 / 异常
  - 子任务详情

影响文件：
- `services/batch_task_service.py`
- `api/routes.py`
- `frontend/index.html`
- `frontend/js/library.js`

验收标准：
- 检测记录页可导出当前任务报告。
- 报告可以脱离页面单独查看。

优先级：
- P1

## 第四阶段：扩展目标来源

### 5. 接入空间测绘目标导入

目标：
- 批量 URL 不只支持手输和 txt，还支持从空间测绘平台导入。

优先接入：
- FOFA
- Hunter
- Quake

参考来源：
- `D:\development\pycharmPRO\pocsuite3\pocsuite3\plugins\target_from_fofa.py`
- `D:\development\pycharmPRO\pocsuite3\pocsuite3\plugins\target_from_hunter.py`
- `D:\development\pycharmPRO\pocsuite3\pocsuite3\plugins\target_from_quake.py`

建议实现：
- 新增 `services/asset_source_service.py`
- 统一目标导入接口
- 前端批量 URL 区新增“从平台导入”

影响文件：
- `services/asset_source_service.py`（新增）
- `api/routes.py`
- `frontend/index.html`
- `frontend/js/library.js`

验收标准：
- 用户可在批量检测页面直接导入目标列表。
- 导入结果可直接进入当前批量任务创建流程。

优先级：
- P1

## 第五阶段：增强请求执行底座

### 6. 新增统一 HTTP Runtime

目标：
- 给生成出来的 Python POC 提供更稳定、更统一的请求层。

建议实现：
- 新增 `services/http_runtime.py`
- 能力包括：
  - 默认超时
  - session 复用
  - 统一请求头
  - SSL / 重定向统一策略
  - 原始 HTTP 报文重放能力预留

参考来源：
- `D:\development\pycharmPRO\pocsuite3\pocsuite3\lib\request\patch\hook_request.py`
- `D:\development\pycharmPRO\pocsuite3\pocsuite3\lib\request\patch\session_reuse.py`
- `D:\development\pycharmPRO\pocsuite3\pocsuite3\lib\request\patch\add_httpraw.py`

影响文件：
- `services/http_runtime.py`（新增）
- `services/poc_library_service.py`
- `prompts/poc_generation.yaml`

验收标准：
- 新生成 POC 可优先使用平台 HTTP helper。
- 旧脚本继续兼容现有 `requests` 方式。
- 请求超时、异常、会话复用行为更稳定。

优先级：
- P2

### 7. 轻量调整 Prompt，引导优先使用平台 HTTP Helper

目标：
- 让模型更稳定地产出适合平台运行的 POC。

调整方向：
- 优先调用平台封装的 HTTP helper。
- 避免散乱的直接 `requests` 调用。
- 复杂 HTTP 利用可调用平台 raw request helper。

影响文件：
- `prompts/poc_generation.yaml`

验收标准：
- 新生成 POC 更统一，更利于执行层接管。

优先级：
- P2

---

## 第六阶段：开放第二类执行模式

### 8. 支持 `url_with_params`

目标：
- 在不破坏 URL-only 主路径的前提下，支持用户手工补充参数执行。

建议实现：
- 前端默认只显示 URL 输入框
- 当 `execution_mode = url_with_params` 时，才展开附加参数区域
- 参数来自 `input_schema`

可支持的输入类型：
- `text`
- `textarea`
- `password`
- `select`
- `json`

影响文件：
- `frontend/index.html`
- `frontend/js/library.js`
- `services/poc_library_service.py`
- `models/schemas.py`
- `api/routes.py`

验收标准：
- URL-only POC 体验完全不变。
- 参数型 POC 可以按 schema 动态渲染输入区。
- 执行层能接收并传递附加参数。

优先级：
- P2

---

## 第七阶段：批量能力与三种模式兼容

### 9. 明确批量执行兼容策略

目标：
- 保证批量检测仍然可控，不因参数型 POC 变得混乱。

建议策略：
- `url_only`
  - 允许普通批量执行
- `url_with_params`
  - 第一版仅允许单条执行
  - 后续可扩展“批量参数模板”
- `manual_guide`
  - 不允许批量执行

影响文件：
- `services/batch_task_service.py`
- `frontend/js/library.js`
- `frontend/index.html`

验收标准：
- 批量检测行为规则明确
- 不会因第二类 POC 把批量逻辑拖垮

优先级：
- P2

---

## 建议执行顺序

1. 定义统一执行模式元数据
2. 接入 Interactsh / CEye
3. 调整 Prompt 支持 OOB 验证
4. 增加批量检测报告导出
5. 接入空间测绘目标导入
6. 新增统一 HTTP Runtime
7. 调整 Prompt 优先使用 HTTP helper
8. 支持 `url_with_params`
9. 明确批量执行兼容策略

---

## 当前不建议提前做

- 直接把整套 `pocsuite3` 控制器或 POCBase 搬进来
- 一开始就做复杂的参数型批量检测
- 一开始就把前端改成大量动态表单
- 在 OOB 能力接入前就修改大模型大量策略

---

## 阶段目标总结

### 第一目标
- 不改变当前“输入 URL 即可验证”的核心路径

### 第二目标
- 让平台支持无回显漏洞验证

### 第三目标
- 让结果可导出、目标可导入、请求层更稳定

### 第四目标
- 最后再放开 `url_with_params`，进入三种执行模式并存阶段
