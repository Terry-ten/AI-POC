## 发现的问题

### 1. `/api/pocs/statistics` 和 `/api/pocs/vuln-types` 当前不可用

测试方法：
- 本地 TestClient 请求：
  - `GET /api/pocs/statistics`
  - `GET /api/pocs/vuln-types`

结果：
- 均返回 `422`

原因：
- 被前面的 `/api/pocs/{poc_id}` 路由抢先匹配
- `statistics` / `vuln-types` 被当成了 `poc_id`

结论：
- 这是一个真实后端路由顺序问题，需要修

### 2. 批量任务取消仍不够优雅

测试方法：
- 创建一个会 `sleep(2)` 的临时 POC
- 创建批量任务后立即取消

结果：
- 任务主状态变成 `cancelled`
- 未启动项被取消
- 已经启动的子任务短时间内仍显示 `running`

结论：
- 取消逻辑可用
- 但“正在执行中的子任务”还没有做到真正中断

### 3. Nuclei 任务创建在 TestClient 环境下未观察到后台推进

测试方法：
- 本地 TestClient 创建真实 Nuclei 任务
- 轮询 `/api/batch-tasks/{id}`

结果：
- 任务可创建
- 但在 TestClient 轮询窗口内未推进，子任务保持 `pending`

结论：
- 接口创建没问题
- 这更像是测试环境下后台线程/外部进程观察限制
- 仍建议在真实运行中的服务里手工验一次

### 4. Nuclei 状态接口的 `version` 字段不够干净

测试方法：
- 本地 TestClient `GET /api/nuclei/status`

结果：
- `version` 返回的是整段 stdout，而不是纯版本号

结论：
- 功能不影响使用
- 但展示和接口语义不够干净，建议后续清理
